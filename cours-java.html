☕ Chapitre 01 · Fondamentaux

Syntaxe Java & Variables

Avant de toucher au Web, il faut comprendre la syntaxe de Java. Ce chapitre t'enseigne les briques fondamentales : déclarer des données, écrire des fonctions, contrôler le flux d'un programme.

📖 ~20 min💻 4 exemples🧩 1 quiz
Section 1.1

Qu'est-ce que Java ?

Java est un langage de programmation créé en 1995. C'est l'un des langages les plus utilisés au monde, notamment dans les grandes entreprises pour créer des applications Web, des systèmes bancaires, des applications Android et bien plus encore.

La grande force de Java, c'est sa devise : "Write once, run anywhere" — tu écris ton code une seule fois, et il fonctionne sur n'importe quel ordinateur. C'est possible grâce à la JVM (Java Virtual Machine), une sorte de traducteur universel installé sur chaque machine.

💡 Analogie

Imagine que tu veux envoyer un message dans différents pays. Plutôt que d'apprendre toutes les langues, tu engages un traducteur universel (la JVM) qui traduit ton message pour chaque pays. Ton code Java, c'est le message que tu écris une seule fois.

Java est aussi un langage fortement typé. Cela signifie que chaque donnée doit avoir un type bien défini. On doit dire à Java si c'est un nombre, du texte, un vrai/faux, etc. Cela peut sembler contraignant au début, mais ça évite énormément d'erreurs — le compilateur te prévient avant même d'exécuter le code.

Section 1.2

Les Variables et les Types de données

Une variable, c'est une boîte qui stocke une information. Quand tu crées une variable en Java, tu dois dire trois choses : le type de l'information, le nom de la boîte, et la valeur qu'elle contient.

💡 Analogie

C'est comme remplir une fiche d'inscription. Tu as une case "Prénom" (String), une case "Âge" (int), une case "Inscrit ?" (boolean). Chaque case a un type précis : on ne met pas un numéro dans la case prénom, sinon l'inscription est refusée.

🔢
Types primitifs
Valeurs simples directement stockées. Commencent par une minuscule : int, double…
📦
Types objets
Référence vers des objets en mémoire. Ont des méthodes. Commencent par une MAJUSCULE.

intPrimitifNombres entiersint age = 21;

booleanPrimitifVrai ou fauxboolean actif = true;

Variables.java
// =============================================
// SYNTAXE : TYPE nomVariable = valeur;
// =============================================

// --- Types primitifs ---
int     age      = 21;
double  prix     = 15000.50;   // Prix en FCFA
boolean connecte = true;        // true ou false UNIQUEMENT

// --- Types objets ---
String  prenom   = "Mouhamed"; // Texte entre guillemets doubles
String  email    = "m@gmail.com";

// --- Afficher des valeurs ---
System.out.println(prenom);            // Affiche : Mouhamed
System.out.println("Âge : " + age);    // Affiche : Âge : 21
System.out.println("Prix : " + prix);  // Affiche : Prix : 15000.5

// --- Les Strings sont des OBJETS et ont des méthodes ---
int     longueur  = prenom.length();          // 8
String  majuscule = prenom.toUpperCase();     // "MOUHAMED"
String  minuscule = prenom.toLowerCase();     // "mouhamed"
boolean aArobase  = email.contains("@");     // true
boolean vide      = prenom.isEmpty();         // false
String  sansEsp   = prenom.trim();            // supprime espaces début/fin
String [] parties  = email.split("@");        // ["m", "gmail.com"]
⚠️ Piège fréquent chez les débutants

Les variables et leurs valeurs sont indépendantes après la copie. Si tu fais int a = 5; int b = a; puis a = 10;, la variable b reste à 5. Chaque variable a sa propre case mémoire (pour les types primitifs).

Section 1.3

Les Méthodes — les actions de Java

Une méthode est un bloc de code réutilisable qui effectue une action précise. Tu lui donnes des informations (les paramètres), elle travaille, et te donne un résultat (la valeur de retour).

💡 Analogie

Une méthode, c'est comme une recette de cuisine. Tu lui donnes des ingrédients (les paramètres), elle exécute les étapes, et te donne un plat (le résultat). La recette "faire du thiéboudienne" prend du riz et du poisson en entrée, et retourne un plat en sortie. Si tu changes les ingrédients, tu obtiens un plat différent.

Chaque méthode en Java a une signature qui définit :

  • La visibilité — qui peut appeler cette méthode (public = tout le monde, private = seulement cette classe)
  • Le type de retour — le type de la valeur retournée (String, int… ou void si rien)
  • Le nom — comment l'appeler (convention : verbe en camelCase)
  • Les paramètres — les informations reçues (type + nom, séparés par virgules)
Methodes.java
// STRUCTURE :
// [visibilité] [type_retour] nomMethode([type param, ...]) {
//     instructions;
//     return résultat; // obligatoire si type_retour != void
// }

// ✅ Méthode qui retourne une String
public String saluer(String prenom) {
    String msg = "Salaam alekoum, " + prenom + " !";
    return msg;   // On retourne le résultat
}

// ✅ Méthode qui retourne un int (calcul)
public int calculerTotal(int prix, int quantite) {
    return prix * quantite;   // On peut return directement
}

// ✅ void = la méthode ne retourne rien (elle fait une action)
public void afficherPrix(double prix) {
    System.out.println("Prix : " + prix + " FCFA");
    // Pas de "return" ici
}

// ✅ boolean : méthode de validation (très utilisée en API Web)
public boolean emailValide(String email) {
    // Un email valide doit contenir "@" et ne pas être null
    return email != null
        && email.contains("@")
        && email.contains(".");
}

// --- Appels ---
String  msg    = saluer("Fatou");         // "Salaam alekoum, Fatou !"
int     total  = calculerTotal(5000, 3);   // 15000
boolean valide = emailValide("a@b.fr");   // true
afficherPrix(25000.0);                      // Affiche "Prix : 25000.0 FCFA"
Section 1.4

Conditions et Boucles

Les conditions permettent à ton programme de prendre des décisions selon les données. Les boucles permettent de répéter une action. Ces deux outils sont quotidiens dans une API Web : valider des données et traiter des listes.

ConditionsBoucles.java
// ===== CONDITIONS =====

String role = "ADMIN";

// if / else if / else
if (role.equals("ADMIN")) {
    System.out.println("Accès complet");
} else if (role.equals("USER")) {
    System.out.println("Accès limité");
} else {
    System.out.println("Accès refusé");
}

// ⚠️ RÈGLE D'OR : Pour comparer des Strings, TOUJOURS .equals()
// role == "ADMIN"        ← FAUX, compare les adresses mémoire !
// role.equals("ADMIN")   ← CORRECT, compare le contenu ✓

// Opérateurs de comparaison pour les nombres
int stock = 0;
if (stock > 0)  System.out.println("En stock");
if (stock == 0) System.out.println("Rupture de stock");
if (stock <= 5) System.out.println("Stock faible");

// Opérateurs logiques : && (ET), || (OU), ! (NON)
boolean majeur = age >= 18;
boolean admin  = role.equals("ADMIN");
if (majeur && admin) System.out.println("Admin majeur");

// ===== BOUCLES =====

// for classique : pour un nombre connu d'itérations
for (int i = 0; i < 5; i++) {
    System.out.println("Tour " + i);  // 0, 1, 2, 3, 4
}

// for-each : la plus utilisée en Web pour parcourir des listes
List<String> villes = List.of("Dakar", "Saint-Louis", "Thiès");

for (String ville : villes) {
    System.out.println("Ville : " + ville);
}
// Affiche : Ville : Dakar / Ville : Saint-Louis / Ville : Thiès
💡 Pourquoi ces notions sont importantes pour le Web ?

Dans une API, tu reçois des données d'utilisateurs que tu dois valider (conditions) avant de les enregistrer, et tu dois souvent parcourir des listes de produits ou d'utilisateurs (boucles) avant de les retourner en JSON.

🧩 Vérifie ta compréhension
En Java, quel est le bon moyen de vérifier si deux Strings ont le même contenu ?
📌 Points clés — Chapitre 01
  • Java est fortement typé : chaque variable doit avoir un type déclaré explicitement
  • Types primitifs (int, double, boolean) vs types objets (String, Integer…)
  • Une méthode a toujours un type de retour — void si elle ne retourne rien
  • Comparer des Strings avec .equals(), jamais avec ==
  • La boucle for-each est la plus utilisée pour parcourir les listes en Web
  • System.out.println() affiche du texte dans la console

Classes & Objets — Modéliser le monde réel

Java est un langage orienté objet. Tout est organisé autour de classes et d'objets. Ce chapitre t'explique comment modéliser des entités du monde réel (utilisateur, produit, commande) en Java.

📖 ~25 min💻 3 exemples🧩 Quiz
Section 2.1

Qu'est-ce qu'une Classe ?

Une classe est un modèle (un "plan de construction") qui décrit les caractéristiques et les comportements d'un type d'objet. Une fois la classe définie, on peut créer autant d'objets (ou "instances") de ce type qu'on veut.

💡 Analogie — Le moule à gâteaux

Imagine un moule à gâteau en forme d'étoile. Le moule est la classe — il définit la forme. Chaque gâteau fabriqué avec ce moule est un objet. Tous ont la même forme (étoile), mais peuvent avoir des parfums différents (vanille, chocolat…). La classe = le moule. L'objet = le gâteau créé.

En développement Web, les classes servent à modéliser les entités du domaine. Par exemple, pour une boutique en ligne :

👤
Utilisateur
id, nom, email, motDePasse, role
📦
Produit
id, nom, prix, stock, categorie
🛒
Commande
id, date, montant, statut, client
🏪
Boutique
id, nom, adresse, ville, telephone
Section 2.2

Créer une Classe complète

Une classe Java se compose de trois éléments : les attributs (les données), le constructeur (comment créer un objet) et les méthodes (getters, setters, comportements). Voici une classe complète commentée ligne par ligne :

Utilisateur.java
// Convention : nom de classe en PascalCase (1ère lettre majuscule)
// Chaque classe = un fichier .java séparé portant le même nom
public class Utilisateur {

    // ================================================
    // 1. ATTRIBUTS (les données de l'objet)
    // "private" = seule cette classe y accède directement
    // C'est la règle d'encapsulation — TOUJOURS private
    // ================================================
    private Long    id;
    private String  nom;
    private String  email;
    private String  motDePasse;
    private String  role;
    private boolean actif;

    // ================================================
    // 2. CONSTRUCTEUR
    // Même nom que la classe, SANS type de retour
    // Appelé avec "new Utilisateur(...)"
    // ================================================
    public Utilisateur(String nom, String email, String motDePasse) {
        this.nom        = nom;        // "this.nom" = l'attribut
        this.email      = email;      // "nom" seul = le paramètre reçu
        this.motDePasse = motDePasse;
        this.role       = "USER";   // Valeur par défaut
        this.actif      = true;    // Actif par défaut
    }

    // ================================================
    // 3. GETTERS — pour LIRE les attributs
    // Spring Boot les utilise pour créer le JSON automatiquement
    // Convention : get + NomAttribut (avec majuscule)
    // ================================================
    public Long    getId()        { return id; }
    public String  getNom()       { return nom; }
    public String  getEmail()     { return email; }
    public String  getRole()      { return role; }
    public boolean isActif()      { return actif; }
    // ⚠️ Pour un boolean : "isActif()" et non "getActif()" !

    // ================================================
    // 4. SETTERS — pour MODIFIER les attributs
    // Permettent de valider avant de modifier
    // Convention : set + NomAttribut (avec majuscule)
    // ================================================
    public void setNom(String nom) {
        if (nom == null || nom.isBlank()) {
            throw new IllegalArgumentException("Le nom ne peut pas être vide");
        }
        this.nom = nom;
    }
    public void setEmail(String email)     { this.email = email; }
    public void setRole(String role)       { this.role = role; }
    public void setActif(boolean actif)    { this.actif = actif; }

    // 5. MÉTHODE UTILITAIRE — comportement de l'objet
    public boolean estAdmin() {
        return "ADMIN".equals(this.role);
    }
}

// ===============================================
// UTILISATION — créer et manipuler des objets
// ===============================================
Utilisateur u1 = new Utilisateur("Mouhamed", "m@gmail.com", "pass123");
Utilisateur u2 = new Utilisateur("Fatou",    "f@yahoo.fr",  "secret");

System.out.println(u1.getNom());    // Mouhamed
System.out.println(u2.getEmail());  // f@yahoo.fr
System.out.println(u1.estAdmin());  // false (role = USER par défaut)

u1.setRole("ADMIN");
System.out.println(u1.estAdmin());  // true maintenant
Section 2.3

Pourquoi private ? — L'Encapsulation

L'encapsulation est le principe qui consiste à cacher les données internes d'un objet et à n'y donner accès que via des méthodes contrôlées. C'est une règle fondamentale en Java et dans toute la POO.

💡 Analogie — Le distributeur automatique

Dans un distributeur automatique, les billets sont protégés dans un coffre (attributs private). Tu ne peux pas les prendre directement avec ta main. Tu utilises les boutons et le clavier (méthodes public) pour interagir. Le distributeur vérifie ton code PIN avant de donner quoi que ce soit — exactement comme un setter qui valide avant de modifier.

❌ Sans encapsulation — attributs public

Si les attributs étaient public, n'importe qui dans le code pourrait écrire :
utilisateur.age = -999; ou utilisateur.email = null;
Aucune validation possible. Les données corrompues arrivent directement en base de données.

✅ Avec encapsulation — attributs private + setters

Le setter valide avant d'accepter la modification. Si quelqu'un essaie de mettre un âge négatif ou un email null, on lance une exception immédiatement — avant que la donnée ne soit sauvegardée. Le code échoue tôt et de façon contrôlée.

📌 Points clés — Chapitre 02
  • Classe = modèle. Objet = instance créée depuis ce modèle avec new
  • Les attributs sont private : on ne peut pas y accéder directement de l'extérieur
  • Le constructeur porte le même nom que la classe et crée l'objet
  • this.attribut = l'attribut de la classe (pour distinguer du paramètre)
  • Les getters lisent (getNom()), les setters modifient (setNom())
  • Pour un boolean, le getter s'appelle isActif() et non getActif()
  • Spring Boot lit les getters automatiquement pour convertir en JSON

Héritage & Interfaces

Comment réutiliser du code, définir des contrats entre classes, et comprendre le pattern Interface → Implémentation qui est au cœur de chaque application Spring Boot.

📖 ~20 min💻 3 exemples
Section 3.1

L'Héritage — réutiliser du code avec extends

L'héritage permet à une classe "enfant" de récupérer automatiquement tous les attributs et méthodes d'une classe "parente", sans les réécrire. On utilise le mot-clé extends.

💡 Analogie

Un "Admin" et un "Client" sont tous les deux des "Personnes". Ils partagent un nom et un email. Plutôt que de répéter ces champs dans chaque classe, on crée une classe Personne parente, et Admin et Client héritent de tout automatiquement. C'est comme les caractéristiques héréditaires dans une famille.

Heritage.java
// CLASSE PARENTE — contient les données communes
public class Personne {
    protected String nom;    // "protected" = accessible aux classes enfants
    protected String email;

    public Personne(String nom, String email) {
        this.nom   = nom;
        this.email = email;
    }

    public String getNom()   { return nom; }
    public String getEmail() { return email; }

    public String sePresenter() {
        return "Je m'appelle " + nom;
    }
}

// CLASSE ENFANT — hérite de Personne + ajoute ses propres champs
// "extends" signifie "hérite de"
public class Client extends Personne {
    private String adresse;
    private int    nombreCommandes;

    public Client(String nom, String email, String adresse) {
        super(nom, email);   // Appelle le constructeur de Personne
        this.adresse         = adresse;
        this.nombreCommandes = 0;
    }

    public String getAdresse() { return adresse; }

    // @Override = on REDÉFINIT la méthode de la classe parente
    @Override
    public String sePresenter() {
        return super.sePresenter() + ", client fidèle";
    }
}

// UTILISATION
Client c = new Client("Ibrahima", "ibra@gmail.com", "Dakar");
System.out.println(c.getNom());        // "Ibrahima" (hérité de Personne)
System.out.println(c.getAdresse());    // "Dakar" (propre à Client)
System.out.println(c.sePresenter());  // "Je m'appelle Ibrahima, client fidèle"
Section 3.2

Les Interfaces — des contrats obligatoires

Une interface définit une liste de méthodes qu'une classe doit obligatoirement implémenter. C'est un contrat. Elle dit "ce que" doit faire une classe, mais pas "comment". Une classe peut implémenter plusieurs interfaces.

💡 Analogie — La prise électrique

Une prise électrique définit un standard : 220V, forme précise des broches. N'importe quel appareil (implémentation) peut s'y brancher, tant qu'il respecte ce standard. Un chargeur téléphone et un aspirateur respectent le même "contrat de prise", mais fonctionnent complètement différemment à l'intérieur.

ProduitService.java + ProduitServiceImpl.java
// INTERFACE : définit le "contrat" du service produit
// Toute implémentation DOIT fournir ces méthodes
public interface ProduitService {
    List<Produit> findAll();
    Produit       findById(Long id);
    Produit       save(Produit produit);
    void          delete(Long id);
}

// IMPLÉMENTATION : respecte le contrat et définit "comment"
// Convention en Spring Boot : suffixe "Impl"
public class ProduitServiceImpl implements ProduitService {

    @Override   // Obligatoire — signale qu'on respecte le contrat
    public List<Produit> findAll() {
        return produitRepository.findAll();   // Requête en BDD
    }

    @Override
    public Produit findById(Long id) {
        return produitRepository.findById(id)
            .orElseThrow(() -> new RuntimeException("Produit #" + id + " introuvable"));
    }

    @Override
    public Produit save(Produit p) {
        // Règle métier : prix ne peut pas être négatif
        if (p.getPrix() < 0) throw new IllegalArgumentException("Prix invalide");
        return produitRepository.save(p);
    }

    @Override
    public void delete(Long id) {
        produitRepository.deleteById(id);
    }
}
💡 Pourquoi interface + implémentation ?

Si demain tu changes de base de données (MySQL → MongoDB), tu crées une nouvelle implémentation ProduitServiceMongo qui respecte le même contrat. Le Controller et les tests ne changent pas une seule ligne. C'est le principal avantage de ce pattern dans Spring Boot.

📌 Points clés — Chapitre 03
  • extends = héritage, une classe enfant récupère tout de la classe parente
  • super(...) dans le constructeur enfant appelle le constructeur parent
  • protected = accessible à la classe et à ses enfants (mais pas à l'extérieur)
  • @Override indique qu'on redéfinit une méthode héritée ou d'interface
  • Une interface définit un contrat (liste de méthodes sans implémentation)
  • implements = une classe s'engage à implémenter toutes les méthodes de l'interface
  • En Spring Boot : toujours Interface + Impl pour les Services
🏷️ Chapitre 04 · Fondamentaux

Les Annotations Java

Les annotations sont des métadonnées que tu places sur ton code pour donner des instructions au framework. Elles commencent par @. Tu en utiliseras des dizaines en Spring Boot.

📖 ~15 min💻 2 exemples
Section 4.1

C'est quoi une annotation ?

Une annotation est une instruction donnée au compilateur ou au framework. Elle commence par @ et se place juste au-dessus de la classe, méthode ou attribut qu'elle concerne. Tu n'as pas à implémenter les annotations — tu les utilises seulement, et Spring Boot fait le reste.

💡 Analogie — Les étiquettes de bagages

À l'aéroport, tu colles des étiquettes sur tes valises. L'étiquette "Fragile" dit aux bagagistes de faire attention. L'étiquette avec la destination dit à quel avion envoyer la valise. Tu n'expliques pas comment transporter la valise — tu donnes juste une instruction et le système s'en occupe. Les annotations en Java, c'est exactement ça.

Section 4.2

Exemple — Un Controller complet annoté

UserController.java
@RestController               // "Cette classe répond à des requêtes HTTP"
@RequestMapping("/api/users") // "Toutes les URLs commencent par /api/users"
public class UserController {

    private final UserService userService;

    // Injection par constructeur (recommandée vs @Autowired sur l'attribut)
    // Spring détecte automatiquement et fournit un UserService
    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping            // GET /api/users → retourne tous les users

Le protocole HTTP — la langue du Web

Le serveur prépare et répond (réponse HTTP) avec le plat (JSON) ou un message d'erreur (404).

Un Servlet — la classe qui reçoit les requêtes

Un Servlet est une classe Java qui reçoit les requêtes HTTP et envoie des réponses. C'est la technologie fondamentale sur laquelle Spring Boot est construit. Comprendre les Servlets te permet de comprendre ce que Spring fait automatiquement pour toi.

ProduitServlet.java
import jakarta.servlet.*;
import jakarta.servlet.http.*;

@WebServlet("/api/produits")  // Ce Servlet répond à l'URL /api/produits
public class ProduitServlet extends HttpServlet {

    // HttpServletRequest  = tout ce que le CLIENT envoie
    // HttpServletResponse = tout ce qu'on retourne au CLIENT

    protected void doGet(HttpServletRequest req, HttpServletResponse res)
            throws IOException {

        // Configurer la réponse
        res.setContentType("application/json");
        res.setCharacterEncoding("UTF-8");
        res.setStatus(200);

        // Écrire le JSON de réponse manuellement (Spring le fait auto !)
        PrintWriter out = res.getWriter();
        out.println("[{\"id\":1,\"nom\":\"Riz Brisé\",\"prix\":25000}]");

Pourquoi Spring Boot ?

Section 6.2

L'Architecture 3 Couches

Spring Boot impose une architecture en 3 couches qui sépare clairement les responsabilités. C'est la convention adoptée dans toute l'industrie Java. Chaque couche ne connaît que la couche du dessous.

💡 Analogie — Un restaurant bien organisé

Le Controller = le serveur en salle : reçoit les commandes des clients et les transmet en cuisine. Il ne cuisine jamais. Le Service = le chef cuisinier : prépare les plats selon les recettes (règles métier). Il ne sert pas les tables. Le Repository = le garde-manger : sait où sont les ingrédients et les récupère. Il ne cuisine pas.

Architecture 3 couches complète
// ============================================================
// COUCHE 1 : CONTROLLER — HTTP uniquement, délègue au Service
// Ne contient AUCUNE logique métier
// ============================================================
@RestController
@RequestMapping("/api/produits")
public class ProduitController {

    private final ProduitService produitService;

    public ProduitController(ProduitService produitService) {

    public ResponseEntity<List<Produit>> findAll() {

    }

    @GetMapping("/{id}")

    }

    public ResponseEntity<Produit> create(@RequestBody Produit p) {

    public ResponseEntity<Produit> update(@PathVariable Long id,
                                              @RequestBody Produit p) {

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {

🗄️ Chapitre 08 · Base de données

JDBC — Parler à MySQL depuis Java

JDBC (Java Database Connectivity) est l'API standard Java pour se connecter à une base de données. Comprendre JDBC te permet de savoir ce que Spring Data JPA fait automatiquement.

📖 ~25 min💻 3 exemples
Section 8.1

Comment fonctionne JDBC ?

JDBC définit un ensemble d'interfaces standard pour parler aux bases de données. Peu importe si tu utilises MySQL, PostgreSQL ou Oracle — le code Java reste identique, seul le "driver" (pilote) change.

1

Ouvrir une connexion

Se connecter à la base de données avec l'URL, l'utilisateur et le mot de passe via DriverManager.getConnection().

2

Préparer la requête

Créer un PreparedStatement avec la requête SQL paramétrée (les ? sont les paramètres sécurisés).

3

Exécuter et lire

Exécuter la requête et parcourir le ResultSet ligne par ligne pour construire les objets Java.

4

Fermer les ressources

Toujours fermer la connexion pour éviter les fuites mémoire. Le try-with-resources le fait automatiquement.

ProduitDAO.java
import java.sql.*;
import java.util.*;

public class ProduitDAO {

    private static final String URL  = "jdbc:mysql://localhost:3306/maboutique";
    private static final String USER = "root";
    private static final String PASS = "motdepasse";

    private Connection connect() throws SQLException {
        return DriverManager.getConnection(URL, USER, PASS);
    }

    // ✅ SELECT — lire tous les produits
    public List<Produit> findAll() {
        List<Produit> liste = new ArrayList<>();
        String sql = "SELECT id, nom, prix, categorie FROM produits";

        // try-with-resources : ferme automatiquement conn, stmt, rs à la fin
        try (Connection conn = connect();
             Statement  stmt = conn.createStatement();
             ResultSet  rs   = stmt.executeQuery(sql)) {

            while (rs.next()) {   // Parcourir chaque ligne du résultat
                Produit p = new Produit();
                p.setId(rs.getLong("id"));
                p.setNom(rs.getString("nom"));
                p.setPrix(rs.getDouble("prix"));
                p.setCategorie(rs.getString("categorie"));
                liste.add(p);
            }
        } catch (SQLException e) {
            throw new RuntimeException("Erreur BDD : " + e.getMessage(), e);
        }
        return liste;
    }

    // ✅ INSERT avec PreparedStatement — OBLIGATOIRE pour la sécurité
    // Les "?" empêchent les injections SQL
    public Produit save(Produit p) {
        String sql = "INSERT INTO produits (nom, prix, categorie) VALUES (?, ?, ?)";

        try (Connection        conn = connect();
             PreparedStatement stmt = conn.prepareStatement(sql, Statement.RETURN_GENERATED_KEYS)) {

            stmt.setString(1, p.getNom());       // 1er "?"
            stmt.setDouble(2, p.getPrix());      // 2e "?"
            stmt.setString(3, p.getCategorie()); // 3e "?"
            stmt.executeUpdate();

            // Récupérer l'ID auto-généré par MySQL
            ResultSet keys = stmt.getGeneratedKeys();
            if (keys.next()) p.setId(keys.getLong(1));

        } catch (SQLException e) {
            throw new RuntimeException("Erreur insertion : " + e.getMessage(), e);
        }
        return p;
    }

    // ✅ DELETE
    public void delete(Long id) {
        try (Connection        conn = connect();
             PreparedStatement stmt = conn.prepareStatement("DELETE FROM produits WHERE id = ?")) {
            stmt.setLong(1, id);
            stmt.executeUpdate();
        } catch (SQLException e) {
            throw new RuntimeException("Erreur suppression", e);
        }
    }
}
❌ Injection SQL — Ne jamais faire ça !

String sql = "SELECT * WHERE nom = '" + nom + "'";
Si l'utilisateur tape ' OR '1'='1, il accède à toutes les données. Toujours utiliser PreparedStatement avec des ?. C'est non-négociable.

📌 Points clés — Chapitre 08
  • JDBC = l'API Java standard pour communiquer avec les bases de données
  • Le flux : Connection → PreparedStatement → ResultSet
  • Toujours utiliser try-with-resources pour fermer automatiquement les connexions
  • PreparedStatement avec ? protège contre les injections SQL (sécurité critique)
  • rs.next() avance ligne par ligne dans les résultats SQL
  • En pratique Spring Boot + JPA remplace JDBC, mais il faut le comprendre

JPA & Hibernate — L'ORM de Java

JPA (Java Persistence API) et son implémentation Hibernate transforment tes classes Java en tables SQL. Plus besoin d'écrire du SQL pour les opérations courantes.

📖 ~30 min💻 4 exemples
Section 9.1

Qu'est-ce qu'un ORM ?

Un ORM (Object-Relational Mapping) fait le lien entre le monde objet (Java) et le monde relationnel (SQL). Il mappe tes classes sur des tables et génère les requêtes automatiquement.

💡 Analogie — Le traducteur automatique

Tu parles Java, la base de données parle SQL. L'ORM est un traducteur simultané. Tu lui dis "sauvegarde cet objet Produit" en Java, il traduit et exécute INSERT INTO produits VALUES (...) en SQL. Tu n'écris plus de SQL pour les opérations courantes.

Section 9.2

Les Entités JPA — mapper les classes sur les tables

Produit.java (Entité JPA complète)
@Entity                        // ← "Cette classe = une table SQL"
@Table(name = "produits")      // ← Nom de la table (optionnel)
public class Produit {

    @Id                         // ← Cet attribut est la clé primaire
    @GeneratedValue(strategy = GenerationType.IDENTITY) // AUTO_INCREMENT
    private Long id;

    @Column(nullable = false, length = 100) // NOT NULL, VARCHAR(100)
    private String nom;

    @Column(nullable = false)
    private Double prix;

    @Column(length = 50)
    private String categorie;

    @Column(name = "en_stock")
    private Boolean enStock = true;

    @Column(name = "date_creation")
    private LocalDateTime dateCreation;

    // RELATION Many-to-One : Plusieurs produits → Une boutique
    // En SQL : colonne "boutique_id" dans la table produits
    @ManyToOne
    @JoinColumn(name = "boutique_id")
    private Boutique boutique;

    // ⚠️ Constructeur vide OBLIGATOIRE pour JPA
    public Produit() {}

    public Produit(String nom, Double prix, String categorie) {
        this.nom          = nom;
        this.prix         = prix;
        this.categorie    = categorie;
        this.dateCreation = LocalDateTime.now();
    }

    // Getters et Setters...
}
